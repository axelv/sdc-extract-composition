More radical ideas...

Option A: Mapping chips in format row

Each mapping becomes its own editable chip. No separate panel.

Format: : Suffix: ! | Map: Nee → × Ja → × Misschien → ×
Onderzoek door Anesthesist || map.
Idea: Mappings ARE the format row - no separate UI needed

Option B: Edit on the pill directly

Click the pill in the editor - mappings appear attached to it.

Format: : Map : Suffix: !
Onderzoek door Anesthesist || map
Nee →
Ja →
.
Idea: The pill owns its config - edit where you see it

Option C: Skip UI - use answer display values

The Questionnaire already has display values for each answerOption. Just use those by default - no mapping needed in most cases.

Format: : Suffix: ! — no map filter needed, uses answerOption.display
Onderzoek door Anesthesist.
Idea: Most mappings = original display. Only add Map filter for overrides.

Option D: Ultra-compact inline table

If we need mappings, show them as compact as possible - single row.

Format: : Map : Suffix: !
Nee→No Ja→Yes Misschien→?
Onderzoek door Anesthesist || map.
Idea: Show state compactly, click "edit" only when needed

Maybe the real question...
How often do users actually need custom mappings? If the Questionnaire's answerOption.display is usually fine, we could:
  • Default to using display values (no Map needed)
  • Only show Map filter as an "override" option
  • Make it rare enough that a popover/modal is acceptable when needed
